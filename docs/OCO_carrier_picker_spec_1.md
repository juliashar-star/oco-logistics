# Подбор перевозчика — спецификация (версия 1, эвристики)

> Статус: черновик для согласования. Версия 1 работает на открытых данных
> (без накопленной статистики). По мере наполнения Carrier Score реальные
> данные (фактический срок, % возвратов) подмешиваются поверх эвристики —
> интерфейс не меняется.

## Принцип: два слоя

Маркетплейсных категорий — сотни, и писать правило выбора под каждую
неподдерживаемо и ложно-точно (женская/мужская/детская одежда логистически
идентичны). Поэтому:

- **Слой 1 — категории.** Большой «маркетплейсный» список, из которого
  продавец выбирает (чтобы любой нашёл себя — вау + поисковый трафик).
- **Слой 2 — логистические профили.** Каждая категория привязана к одному
  из 7 профилей. Именно профиль (плюс регион, приоритет, способ) управляет
  рекомендацией.

Продавец видит сотню знакомых категорий — движок оперирует семью профилями.

## Природа профилей (читать первым)

P1–P7 — **бизнес-абстракция OCO**: проекция маркетплейс-категорий товара на логистику.
Перевозчики так **не** мыслят — у них классификация по физике отправления
(тарифные классы по весу/плотности; стандарт/негабарит по размеру).
Категория товара перевозчику безразлична — важны вес, размер, плотность, режим.

Ценность профилей для продавца: он думает категорией товара («Мебель», «Цветы»),
а не классом груза. Это сознательное упрощение OCO, **не** стандарт APIShip.

- **P1–P4** — границы качественные (по типу товара / категории).
- **P5–P6** — добавляются числовые пороги (вес, габарит), см. ниже.
- **P7** — по режиму (скоропорт / температура), не по весу.

## Слой 2: логистические профили

| # | Профиль | Логистические свойства | Что важно при выборе СД |
|---|---|---|---|
| P1 | Лёгкое-мягкое, частые возвраты | малый вес, не бьётся, высокий % примерок/возвратов | простой возврат, широкая сеть ПВЗ |
| P2 | Хрупкое-компактное | стекло/жидкости, бьётся, небольшой размер | аккуратность, страховка, ПВЗ |
| P3 | Мелкое-ценное | высокая стоимость на единицу, компактное | страховка, надёжность, контроль |
| P4 | Стандартное-небольшое | средний вес/габарит, не хрупкое | цена/скорость, гибкость формата |
| P5 | Тяжёлое/объёмное (не КГТ) | большой вес или объём, но в пределах посылки | лимиты веса, тариф за вес |
| P6 | Крупногабарит / КГТ | паллеты, мебель, крупная техника | грузовые ТК, терминал/адрес |
| P7 | Скоропорт / спецрежим | срок годности, температура, особые требования | скорость, день-в-день, локально |

## Источник числовых границ P5 / P6

Лимиты перевозчиков подтверждены официальными источниками (проверено 06.2026):

- Boxberry: до 15 кг, сторона до 120 см (boxberry.ru)
- СДЭК тариф «Посылка»: до 30 кг (cdek.ru)
- DPD: до 31,5 кг, длинная сторона до 175 см (dpd.ru)
- Почта России: до 20 кг
- Деловые Линии, Байкал, Возовоз, ПЭК: сборные грузы, без практического B2C-лимита

Границы профилей выведены из этих лимитов — **наше допущение**,
подлежащее пересмотру при накоплении статистики Carrier Score.

### Числовые границы P5 / P6

| Параметр | P5 | P6 |
|---|---|---|
| Вес | 15–30 кг | свыше 30 кг |
| Длинная сторона | до 120 см | свыше 120 см |

### Логика классификации (порядок проверки)

1. Если `weight` отсутствует для категории P5/P6 → `{ profile: null, reason: "weight_required" }`
2. Если `weight > 30` кг **или** длинная сторона `> 120` см → **P6**
3. Если `weight` 15–30 кг (и сторона ≤ 120 см) → **P5**
4. Иначе → классификация по категории (P1–P4, P7) как сейчас

Приоритет габарита: при стороне > 120 см → P6 независимо от веса.
При весе < 5 кг, но стороне 60–120 см → P4 (габарит важнее веса).

### Краевые случаи

> ⚠️ Устарело по состоянию на 2026-07-06 — фильтрация по подключению убрана; теперь показывается весь подходящий рынок с меткой isConnected (см. DECISIONS.md, 2026-07-06).

| Ситуация | Профиль | Reason |
|---|---|---|
| Вес < 5 кг, сторона 60–120 см | P4 | — |
| Вес < 5 кг, сторона > 120 см | P6 | — |
| Категория «Мебель» (P6), вес 8 кг | P6 | категория важнее (КГТ) |
| Нет `weight` для P5/P6-категории | null | `weight_required` |
| Нет подключённых перевозчиков | null | `no_carrier_connected` |

## Правила ранжирования

Вход анкеты: **категория** (→ профиль), **регионы доставки**, **приоритет**
(дешевле / быстрее / надёжнее / меньше возвратов), **способ** (ПВЗ / курьер / оба).
Для P5/P6 дополнительно: **вес** и **длинная сторона** (см.).

### P1–P4: score-карты (регион / способ / приоритет)

Алгоритм:
1. По профилю отобрать СД, у которых профиль в списке поддерживаемых
   (например, P6 КГТ убирает Boxberry с лимитом 15 кг).
2. По региону: «вся РФ / малые города» поднимает Почту и СДЭК; «город» —
   Яндекс, Dostavista.
3. По способу: «только ПВЗ» поднимает Boxberry, СДЭК, 5POST; «курьер» —
   Яндекс, СДЭК, DPD.
4. По приоритету: «дешевле» — Boxberry, Почта, постаматы; «быстрее» —
   Яндекс, СДЭК-экспресс; «надёжнее» — СДЭК, Boxberry.
5. Вернуть топ-2–3 с коротким объяснением «почему».

Объяснимость обязательна — обоснование «почему именно эти» и есть часть ценности.

### P5 / P6 / P7: фиксированный приоритетный список

Для этих профилей — **фиксированный порядок**, не REGION / METHOD / PRIORITY:

| Профиль | Порядок (`providerKey`) |
|---|---|
| P5 | `dpd` → `cdek` → `dellin` → `baikalsr` → `vozovoz` → `rupost` |
| P6 | `dellin` → `baikalsr` → `pecom` → `vozovoz` → `cdek` |
| P7 | `yataxi` → `dostavista` → `logsis` |

Обоснование:
- **P5:** курьерские с трекингом (`dpd`, `cdek`) первыми; грузовые как fallback; почта последней.
- **P6:** специалисты по сборным грузам первыми; `cdek` последним (КГТ — отдельный договор, не стандартный API).
- **P7:** курьерская экспресс день-в-день, порядок по покрытию.

**«Подключён»** = пересечение перевозчика с `listConnections()` продавца.
Отдельная ветка в `rankCarriers()`, которой сейчас нет — добавить при реализации.

### Gap в API (`weight`) — закрыть при кодировании P5/P6

Сейчас route принимает `parcel.weight`, но не передаёт в `rankCarriers()`.
При реализации:
1. Прокинуть `weight` (и `maxSideCm`) в `RankInput`
2. Классификация P5/P6 как pre-step до ранжирования
3. Без веса для P5/P6-категорий — `reason: "weight_required"`, не молчаливый fallback на P4

### Текущее состояние кода (до реализации P5/P6/P7)

В `rank.ts` сейчас **все** профили ранжируются score-картами P1–P4.
Для P5/P6 это даёт артефакт: СДЭК набирает B2C-баллы и обгоняет грузовых.
Правила выше формализованы; перенос в код — следующий шаг после согласования спеки.

## Модель мониторинга актуальности (3 уровня надёжности)

Цель: на сайте только актуальные данные. Полная автоматизация смысловых изменений
правил перевозчиков **недостижима** — лимиты/условия СД не публикуются машинно
(HTML, PDF-регламенты, калькуляторы, разные форматы). Поэтому гибрид:

### Уровень 1 — авто, еженедельно (реализовано)

Drift-скрипт по `listProviders()`: новые / пропавшие перевозчики.
`scripts/check-carrier-drift.ts`, `npm run carrier:drift`.
Exit 1 при пропаже перевозчика из `CARRIER_REGISTRY` (removed).

### Уровень 2 — авто, ежемесячно (запланировано)

Снимок числовых параметров через APIShip-калькулятор: расчёт стоимости
для эталонной посылки по каждому перевозчику. Аномалия (скачок цены / недоступность)
→ флаг на ручную проверку. Реализуемо через API, надёжно. Код позже.

### Уровень 3 — ручной ревизионный цикл, ежеквартально (каркас)

Реестр «источников правды» — официальные страницы лимитов каждого СД,
с датой последней проверки. Скрипт напоминает, что квартал прошёл.
Смысловые изменения правил (новые лимиты, тарифы, услуги) сверяются вручную.

Реестр источников правды (страницы лимитов для квартального ревью):

- Boxberry: boxberry.ru → требования к отправке
- СДЭК: cdek.ru → тарифы для интернет-магазинов
- DPD: dpd.ru → калькулятор / тарифы
- Почта России: pochta.ru → ограничения по весу / габаритам
- Деловые Линии: dellin.ru → крупногабарит
- Байкал Сервис, Возовоз, ПЭК: офиц. сайты, условия сборных грузов

## Публичная часть vs кабинет

- **До регистрации (тизер):** один сценарий (одна категория + один регион),
  топ-1 СД + одна строка обоснования. Вау есть, но это «заголовок».
- **После регистрации:** полный ранжир по всем категориям/регионам продавца,
  развёрнутое обоснование; по мере накопления — реальный Carrier Score
  (срок, возвраты) поверх эвристики и рекомендация в момент создания заказа.
