# Non-functional — что закладываем сейчас, а что потом

Нефункциональные требования (надёжность, безопасность, масштаб). Этот файл говорит Cursor,
**что важно сделать в MVP, а что осознанно отложить**, чтобы не строить инфраструктуру под
нагрузку, которой пока нет.

**Принцип отбора:** в MVP закладываем то, что либо (а) критично для безопасности и целостности
данных, либо (б) дёшево сделать сейчас и больно переделывать потом. Всё про масштаб добавляем
тогда, когда появится реальная нагрузка.

---

## Сейчас (в MVP)

- **Фронтенд, API, бэкенд-логика, база данных** — ядро продукта.
- **Аутентификация и права** — вход по email/паролю, сессии, роли OWNER и OPERATOR; проверка
  прав на сервере, защита входа от перебора паролей.
- **Изоляция данных между компаниями (критично)** — каждый запрос к данным ограничен `companyId`
  текущего пользователя. Данные одной компании недоступны другой. Реализуем через общий помощник
  доступа к данным, который всегда подставляет `companyId`. (Опциональное усиление — Row Level
  Security в PostgreSQL.)
- **Контроль версий (git)** — с первого дня: приватный репозиторий, `.env` вне git, понятная
  история коммитов.
- **Логирование без персональных данных** — понятные сообщения об ошибках для пользователя,
  технические логи без ФИО/телефонов/адресов.
- **Бэкапы базы** — ежедневные, с проверенным восстановлением (обязательно на вехе M4, до первых
  реальных данных).
- **HTTPS** — обязательно в бою.
- **Уважение лимитов APIShip** — не опрашивать статусы слишком часто; устойчивость к сбоям API
  (понятная ошибка + повтор, а не «падение»).
- **Хостинг** — один российский VPS (локальная разработка → деплой на M4).

## Скоро (рано, но не в первый день — добавить по ходу MVP)

- **Простой трекер ошибок** — чтобы видеть сбои на сервере (например, GlitchTip/Sentry-совместимый).
  Ставится быстро, помогает чинить баги у первых клиентов. Не обязателен в день старта.
- **Базовое ограничение частоты запросов** на чувствительных роутах (вход, расчёт), чтобы никто
  не злоупотреблял.
- **Возможность удалить персональные данные получателя** по запросу (152-ФЗ) — на старте простое
  удаление/обезличивание записи.

## Потом (откладываем — добавляем по сигналу)

| Тема | Когда добавлять (сигнал) |
|---|---|
| **CI/CD-пайплайн** | Когда деплой станет частым и ручной начнёт раздражать/ошибаться. |
| **Кэширование (Redis) и CDN** | Когда страницы/расчёты начнут заметно тормозить под реальной нагрузкой. |
| **Балансировка нагрузки, несколько серверов** | Когда один сервер перестанет справляться. |
| **Горизонтальное масштабирование, очереди сообщений** | Когда фоновых задач станет много и они начнут мешать друг другу. |
| **Полноценный мониторинг (Grafana/OpenTelemetry)** | Когда нужно следить за метриками в реальном времени, а не разово. |
| **Отказоустойчивость / резервный сервер (HA)** | Когда простой даже на час станет недопустим для клиентов. |
| **Микросервисы, Kubernetes** | Скорее всего, никогда для MVP; только при серьёзном росте команды и нагрузки. |

---

## Важное правило для Cursor
Не добавляй элементы из блока «Потом» по своей инициативе. Если считаешь, что без чего-то из них
никак уже сейчас, — сначала объясни основателю простыми словами, зачем это и какой сигнал
сработал, и спроси разрешение. По умолчанию держим решение простым и одно-серверным.
