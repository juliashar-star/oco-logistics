# Decision Engine — логика выбора доставки (MVP + развитие)

> **Статус на 23.07.2026:** движок НАПИСАН, экспортирован и покрыт тестами, но до него не
> доходит путь заказа. Единственный вызов `rankQuotes` в коде приложения — легаси-роут
> `/api/shipments/calculate`, которого форма нового заказа не касается: она идёт
> create-draft → offers → submit и получает офферы одного подключённого перевозчика.
> Смысл ранжирования вернётся, когда подключится второй перевозчик (CDEK) — и тогда же
> движок надо перенацелить с котировки APIShip на нейтральный `CarrierOffer`.

Это «мозг» продукта: из вариантов доставки выбрать и ранжировать лучшие по цене, сроку и
качеству перевозчика. Документ задаёт, что строим сейчас и что закладываем на будущее, не строя сейчас.

Движок живёт в `packages/core` и не зависит от интерфейса: чистая функция, вход — список
вариантов и настройки, выход — отсортированный список с метками. От APIShip он пока
ЗАВИСИТ: `rank-quotes.ts` импортирует тип `DeliveryQuote` из `@oco/apiship` и работает на
полях котировки APIShip. Это известный хвост, он уходит при переходе на нейтральный
`CarrierOffer`.

## 1. Факторы и коэффициенты (веса)

Каждый вариант оценивается по трём факторам, каждый нормализуется в диапазон 0..1:
- **Цена** (`cost`) — чем меньше, тем лучше.
- **Срок** (`deliveryDays`) — чем меньше, тем лучше.
- **Качество** (`carrierScore` 0..100) — наш собственный рейтинг перевозчика.

Веса (коэффициенты) — это НАСТРОЙКА, а не зашитые в код числа. По умолчанию, например:
`w_cost = 0.4`, `w_speed = 0.3`, `w_quality = 0.3`. Их должно быть легко изменить (сейчас — в коде, позже — в интерфейсе компании). Значения по
умолчанию лежат в `packages/core/src/decision-weights.ts` и нормализуются к сумме 1.

## 2. Режимы выбора

- **CHEAP (Дёшево)** — минимальная цена.
- **FAST (Быстро)** — минимальный срок.
- **OPTIMAL (Оптимально)** — взвешенная сумма:
  `score = w_cost*(1 - normCost) + w_speed*(1 - normSpeed) + w_quality*(carrierScore/100)`
  где `normCost`/`normSpeed` — цена и срок, приведённые к 0..1 внутри текущего набора вариантов.

> **Важно сегодня:** Carrier Score не считается (см. `docs/DATABASE.md` §4), поэтому всем
> вариантам подставляется нейтральное значение 50. Слагаемое качества выходит одинаковым, вес
> `w_quality` не меняет порядок — «Оптимально» сейчас ранжирует по цене и сроку.

- **MANUAL** — пользователь выбрал вручную.

## 3. Фильтры и сортировка (можно сейчас или вскоре)
Над списком вариантов: фильтр по типу доставки (ПВЗ/курьер), по максимальной цене, по
максимальному сроку, по минимальному Carrier Score, по конкретным перевозчикам; сортировка по
любому из факторов.

## 4. Что строим СЕЙЧАС (MVP)
- Режимы CHEAP / FAST / OPTIMAL с весами по умолчанию.
- Веса оформлены как параметры (не «магические числа»), даже если меняются только в коде.
- Базовая сортировка/фильтр списка вариантов.
- Сохранение ВСЕХ вариантов: на легаси-пути APIShip — в `TariffQuote` вместе с полным
  `rawResponse`, на живом пути — вся выдача офферов в `Shipment.quotedOffers`.

## 5. Что ЗАКЛАДЫВАЕМ в архитектуру, но НЕ строим сейчас (Phase 2)

### 5.1 Правила автовыбора (rules engine)
Список правил, проверяемых сверху вниз; срабатывает первое подходящее, иначе — режим по умолчанию.
- Условия: диапазон веса; максимальные габариты; тип доставки (ПВЗ/курьер); регион; категория
  товара; объявленная ценность.
- Действие: форсировать конкретного перевозчика/услугу ИЛИ применить режим (cheap/fast/optimal)
  с порогом минимального качества (например, «оптимально среди перевозчиков с Carrier Score ≥ 70»).
- Хранение: JSON-конфиг правил на компанию. Интерфейс настройки — позже; структуру данных
  предусмотреть заранее.
- Пример правила: «если вес ≤ 1 кг И габариты ≤ 50×30×10 см И доставка в ПВЗ → перевозчик X».

### 5.2 Автоматический режим (без ручного выбора) для крупных продавцов
Для продавцов с сотнями заказов в день: заказ создаётся сразу по правилам/режиму, без клика.
Требует инфраструктуры из `docs/NONFUNCTIONAL.md` (раздел «Потом»): пакетная обработка, фоновые
задачи/очередь, идемпотентность (защита от дублей), учёт лимитов API перевозчиков, мониторинг. Такие
продавцы, как правило, подключаются через API/импорт, а не через веб-форму.

### 5.3 Коэффициенты второго уровня (когда накопятся данные)
- Carrier Score отдельно по региону и по категории товара.
- Стабильность срока (разброс, а не только среднее).
- Доля повреждений/потерь; штраф за просрочку.

## 6. Архитектурное правило на сейчас
Создание отправления — это отдельная функция-сервис, вызываемая и из интерфейса, и (в будущем)
из API/очереди. Не привязывай логику создания заказа к кнопке в UI — иначе автоматический режим
позже потребует переписывать ядро.

Сегодня это выполнено: путь заказа живёт в `apps/web/lib/shipments`, а не в компоненте формы.
