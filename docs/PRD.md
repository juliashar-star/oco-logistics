# PRD — OCO Logistics (MVP)

Product Requirements Document. Версия 1.0. Объём: только логистика, веб-кабинет, хостинг в РФ.
Это «что и зачем мы строим». Cursor читает этот файл, чтобы понимать продукт целиком.

> **Статус на 23.07.2026:** документ приведён к модели F — селлер везёт по СВОИМ договорам
> с перевозчиками, OCO оркеструет заказ и накапливает знание. Пометки «НЕ ПОСТРОЕНО» в §5 и
> §6 описывают состояние кода на эту дату; требование при этом остаётся в силе.

## 1. Проблема

Независимые российские D2C-бренды (Fashion / Beauty / Wellness / Pet), которые хотят продавать
напрямую, а не только через Wildberries и Ozon, теряются в доставке: у каждой службы (СДЭК,
Почта, 5Post, Boxberry, Яндекс и др.) свои тарифы, сроки и качество. Сравнивать вручную долго,
а главное — **никто не показывает объективное качество доставки** (реальные сроки, возвраты),
только цену. Бренды переплачивают и теряют клиентов из-за плохой доставки, сами того не зная.

## 2. Решение

Веб-кабинет, в котором бренд подключает СВОИХ перевозчиков — по своим договорам — и оформляет
отправления в одном месте, не переключаясь между кабинетами служб. OCO запрашивает предложения
у подключённых перевозчиков по конкретной посылке, сравнивает их по **цене, сроку и собственному
рейтингу качества (Carrier Score)** и предлагает варианты «Быстро / Дёшево / Оптимально». По
перевозчикам, которых бренд ещё не подключил, показываем оценочные тарифы из APIShip — как
справку, а не как канал доставки. Чем больше отправлений — тем умнее рекомендации.
Что из перечисленного уже построено, а что нет — см. §5.

## 3. Главная ценность (и почему это защищено от копирования)

Актив продукта — **не код, а накопленные данные**: по каждому отправлению храним обещанный срок
против фактического, возвраты, отмены, категорию товара, перевозчика, регион. Тарифы видны всем,
а объективное качество доставки не измеряет никто. Через несколько тысяч отправлений это —
уникальный набор данных, который конкурент не сможет повторить быстро.

## 4. Кто пользователь (MVP)

**Селлер-бренд** в Москве и МО: владелец или оператор маленького/среднего бренда, который
отправляет от нескольких до сотен посылок в месяц и хочет: платить за доставку меньше,
выбирать надёжных перевозчиков и тратить на оформление минимум времени.

## 5. Объём MVP

**Входит:**
- Регистрация компании и вход (email + пароль).
- **Подключение своих перевозчиков** в настройках: селлер вводит собственные доступы (сначала
  Яндекс Доставка, затем CDEK, дальше по одному), они шифруются и хранятся у нас.
  **НЕ ПОСТРОЕНО — гейт запуска.** Порядок: Яндекс → CDEK → форма → запуск.
- Получение предложений подключённых перевозчиков по посылке и сравнение их по цене и сроку.
- Выбор варианта: «Быстро», «Дёшево», «Оптимально» (с учётом Carrier Score).
  **НЕ ПОСТРОЕНО:** сегодня селлер выбирает оффер из списка вручную.
- Создание отправления у перевозчика — подтверждение выбранного оффера.
- Трек-номер или ссылка для отслеживания, которую бренд отдаёт покупателю.
  **НЕ ПОСТРОЕНО на живом пути:** трек пишется только в легаси-пути APIShip. Сначала надо
  выяснить, что для этого вообще отдаёт Яндекс.
- Список отправлений с фильтрами и статусами (трекинг).
  **Статусы живых заказов не обновляются:** синхронизация Яндекса существует роутом, но её
  не вызывает ни расписание, ни кнопка в списке.
- Сбор полной статистики по каждому отправлению (обещано/факт, возвраты и т.д.).
- Базовая аналитика: дашборд с метриками. **Счётчик экономии не построен.**
- Carrier Score (рейтинг перевозчиков на основе своих данных).
  **Расчёт не построен** — см. `docs/DATABASE.md` §4.

**НЕ входит (не разрабатывать):**
- Маркетплейс, витрина, корзина для конечных покупателей.
- CRM, рассылки, программа лояльности.
- Маркировка «Честный Знак».
- Мобильное приложение, Telegram-бот (позже).
- Свои склады, фулфилмент, собственная доставка.
- ИИ-рекомендации, прогнозы (на старте — простые правила).
- Биллинг/приём оплаты с клиентов (первые 3 месяца бесплатно; экрана «Тарифы» нет).

## 6. Экраны

1. **Dashboard** — метрики за 7/30 дней (число заказов, расходы), график, кнопка
   «Новый заказ». Для нового пользователя — подсказки онбординга. Счётчика экономии нет,
   см. §5.
2. **Новый заказ** (ядро) — форма посылки (категория, вес, габариты, город, тип выдачи:
   курьер/ПВЗ, данные получателя) → предложения подключённых перевозчиков (цена, срок,
   рейтинг) → выбор варианта → «Создать отправление». Сегодня: офферы одного подключённого
   перевозчика, выбор вручную, срок ОРИЕНТИРОВОЧНЫЙ, трек-номера на выходе нет.
3. **Отправления** — таблица со статусами, фильтрами, поиском, трек-номерами, экспорт в CSV.
   На живом пути колонка трек-номера пустая, а статусы не обновляются — см. §5.
4. **Настройки** — одна страница с четырьмя вкладками: Профиль · Компания (адрес отправителя,
   резервная копия) · Безопасность · Подключение (свои перевозчики). Правила автовыбора —
   не в MVP, см. `docs/DECISION_ENGINE.md` §5.1.
5. **Тарифы (Billing)** — «Бесплатно на период пилота». Экран не создан.
6. **Подбор перевозчика** — справочный экран сравнения перевозчиков между собой, не путь заказа.
7. **Сравнение перевозчиков** — отдельная страница, закрытая переключателем
   `ENABLE_CARRIER_COMPARISON_PAGE`; публичный запуск гейтится вопросами к юристу.

## 7. Метрики успеха MVP

- 5 реальных брендов сделали по 10+ отправлений каждый через OCO.
- Время оформления одного заказа — **меньше 3 минут**.
- NPS пилотных брендов выше 30.
- По каждому отправлению собрана полная статистика (обещано/факт).
- Хотя бы у одного перевозчика рассчитан осмысленный Carrier Score.

## 8. Definition of Done (когда функция считается готовой)

Функция готова, только если:
- Она делает то, что описано в соответствующей User Story (см. USER_STORIES.md).
- Есть простой автотест на ключевую логику (Decision Engine, адаптеры перевозчиков,
  оркестрация заказа).
- Проверены крайние случаи: перевозчик не ответил, пустая выдача вариантов, неверный адрес,
  нулевой вес.
- Персональные данные не утекают в логи, URL и сообщения об ошибках.
- На экране понятные сообщения об ошибках (без технического мусора).
- Тестировщик подтвердил, основатель проверил руками по чек-листу.
- Поведение проверено руками в браузере на живом сценарии. Зелёные гейты этого не заменяют:
  форма юнит-тестами не покрыта.
