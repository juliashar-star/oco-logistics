# APIShip Integration — спецификация (MVP)

> **Статус на 23.07.2026.** Этот документ описывает APIShip как **слой знаний**, а не как
> канал доставки. Живой путь заказа (черновик → офферы → подтверждение) идёт через прямые
> адаптеры перевозчиков (модель F, `packages/core/src/carrier-adapter`), APIShip в нём не
> участвует. За APIShip остаются две роли из ADR от 11.07.2026: сырьё для Carrier Score и
> подстановка ОЦЕНОЧНЫХ тарифов по ещё не подключённым перевозчикам. Обе сегодня в бою
> практически недостижимы — почему, см. §1.

Что именно нам нужно от APIShip и как с ним работать. Весь код APIShip живёт ТОЛЬКО в
`packages/integrations/apiship` за единым интерфейсом.

> ⚠️ Важно для Cursor: **точные адреса запросов (endpoints) и форматы полей бери из официальной
> документации** ([docs.apiship.ru](https://docs.apiship.ru/docs/api/)), а не из памяти.
> Ниже — что нам нужно по смыслу. Если формат поля непонятен — спроси основателя, не выдумывай.

## 1. Авторизация (нет статичного API-ключа)

APIShip **не использует** постоянный ключ в заголовке. Схема такая:

1. **POST `/users/login`** — тело `{ "login": "...", "password": "..." }`, токен в ответе **не передаём**.
2. В ответе поле **`token`** (в старых примерах встречается `accessToken` — поддерживаем оба).
3. Токен передаётся в заголовке **`Authorization: <token>`** (без префикса `Bearer`) во всех остальных запросах.
4. **Срок действия токена в текущей версии API не проверяется** — получаем один раз и переиспользуем.
5. Кэш токена — в `packages/integrations/apiship` (в памяти процесса, ключ: `baseUrl + login`).

Документация: [Получение API токена](https://docs.apiship.ru/docs/api/user-service/get-token/),
[Интеграция по API](https://docs.apiship.ru/docs/api/).

### Переменные окружения (только сервер, `.env`)

| Переменная | Назначение |
|---|---|
| `APISHIP_BASE_URL` | Базовый URL API с `/v1/` на конце |
| `APISHIP_LOGIN` | Логин APIShip (для dev — `test`) |
| `APISHIP_PASSWORD` | Пароль APIShip (для dev — `test`) |
| `APISHIP_ENCRYPTION_KEY` | Ключ шифрования `apishipPasswordEnc` в базе (мин. 32 символа, только `.env`) |

**На фронтенд логин, пароль и токен не попадают никогда** — все вызовы APIShip идут через наш бэкенд.

### Тестовый и боевой контур

| Контур | `APISHIP_BASE_URL` | Учётные данные |
|---|---|---|
| **Разработка (sandbox)** | `http://api.dev.apiship.ru/v1/` | `test` / `test` — публичная песочница |
| **Бой (M4)** | `https://api.apiship.ru/v1/` | логин/пароль из личного кабинета APIShip |

- Разработка — **только тестовый контур**.
- Боевые учётные данные подключаем в конце (веха M4), после «ок» основателя.
- **Что делает код сегодня** (`apps/web/lib/apiship-client-for-company.ts`): сначала берутся
  учётные данные КОМПАНИИ — если заполнены `apishipLogin`, `apishipPasswordEnc` и
  `apishipConnectedAt`, клиент создаётся на них. Пароль расшифровывается ключом
  **`APISHIP_ENCRYPTION_KEY`** из `.env` (минимум 32 символа, не в коде); расшифрованный
  пароль в логи не пишем. `APISHIP_BASE_URL` берётся из `.env` всегда.

- Fallback на `APISHIP_LOGIN` / `APISHIP_PASSWORD` из `.env` — **только при
  `NODE_ENV !== production`**. Если нет ни кредов компании, ни fallback — клиент бросает
  ошибку «APIShip не подключён». Молча пропуска нет.

- ⚠️ **Следствие, важное для планирования.** Форму подключения APIShip удалили 23.07.2026
  (коммит `0d709a1`, хвост ROADMAP 31), а fallback на `.env` в бою выключен по `NODE_ENV`.
  Значит заполнить `apishipLogin` / `apishipPasswordEnc` больше нечем: **в бою APIShip
  работает только у компаний, подключившихся ДО удаления формы, а новый продавец не может
  подключиться никак.** Это касается всех шести потребителей, которые ходят через
  `getApishipClientForCompany`: `/api/shipments/calculate`, `/intervals`, `/points`,
  легаси-создание заказа (`lib/shipments/create-shipment.ts`), синхронизация статусов
  APIShip (`lib/shipments/sync-statuses.ts`) и список подключённых перевозчиков для
  подборщика (`lib/carrier-picker/connected-carriers.ts`). Как именно деградирует каждый
  из шести — НЕ ПРОВЕРЕНО; известно только, что `/calculate` отдаёт 400 по гейту
  `canUseApiship`.

- **Задумано, НЕ ПОСТРОЕНО:** при модели F учётные данные APIShip — это ключи САМОГО OCO
  (один аккаунт на платформу, слой знаний), а не продавца; по-селлерные колонки
  `Company.apiship*` остаются легаси и уйдут вместе с чисткой легаси-маршрутов. Правило
  `.cursor/rules/08` уже описывает мастер-аккаунт как целевую схему — код к ней не приведён.

## 2. Возможности: что нужно по модели F и что осталось от легаси

**Нужно (роль слоя знаний, ADR от 11.07.2026):**

1. **Расчёт тарифов** — `POST /calculator` ([документация](https://docs.apiship.ru/docs/api/calculator/)).
   Нужен для ОЦЕНОЧНЫХ тарифов по перевозчикам, которых продавец ещё не подключил напрямую.
   Показ таких оценок в интерфейсе до сих пор **гейтится непроверенным ToS APIShip** (§6).
2. **Сырьё для Carrier Score** — история статусов и фактические сроки как вход для рейтинга.
   Таблица `CarrierScore` пуста, расчёта нет (см. `DECISION_ENGINE.md`).

**Легаси — на живом пути заказа НЕ используется** (там прямые адаптеры перевозчиков):
создание заказа, этикетка/документы, статусы/трекинг, список ПВЗ, вызов курьера. Код жив и
зелёный, но форма нового заказа туда не ходит; всё это уйдёт вместе с чисткой
легаси-маршрутов. **Новое на этом не строить.**

**Не реализовано:** валидация адреса (ApiDQ).

## 3. Единый интерфейс (модуль `packages/integrations/apiship`)

| Метод | Назначение |
|---|---|
| `getToken()` / внутренний кэш | получить и переиспользовать токен |
| `testConnection()` | проверка логина/пароля |
| `calculate(input)` | массив вариантов доставки |
| `listPoints(city)` | список ПВЗ для выбора в форме заказа |
| `createOrder(input)` | создание отправления — реализовано, но только на ЛЕГАСИ-пути (`apps/web/lib/shipments/create-shipment.ts`) |
| `getStatus(...)` | трекинг — реализовано, но только на ЛЕГАСИ-пути (`apps/web/lib/shipments/sync-statuses.ts`) |

Остальной код вызывает только эти методы и не знает про устройство APIShip.

**На живом пути заказа этот интерфейс не участвует.** Там работает `CarrierAdapter` из
`packages/core/src/carrier-adapter` — прямые адаптеры перевозчиков (модель F).

## 4. Что сохраняем из ответов (это актив)

> **Сегодня весь этот раздел относится к ЛЕГАСИ-пути APIShip.** На живом пути офферы
> сохраняются в `Shipment.quotedOffers`, а `trackNumber` и `labelUrl` не пишутся вообще
> (см. `USER_STORIES.md`: US-3.4 и эпик 4).

- Из расчёта → **все** варианты в `TariffQuote` (даже невыбранные) — легаси-таблица.
- Из создания заказа → `apishipOrderId`, `trackNumber`, плановые цена и срок в `Shipment`.
- Из статусов → события в `TrackingEvent`, фактические даты и возвраты в `Shipment`.

## 5. Устойчивость к сбоям

- APIShip недоступен → понятное сообщение пользователю, приложение не «падает».
- Логи **без ПДн** (без ФИО/телефона/адреса) и без логина/пароля APIShip.
- Уважаем лимиты APIShip (статусы — не чаще N часов).

### Решено 23.07.2026 — вопросы выше сняты

Выбран **Вариант 1: аккаунт APIShip принадлежит самому OCO.** Продавцы APIShip не
подключают — они подключают своих перевозчиков напрямую (модель F), а APIShip остаётся
слоем знаний. Полное решение и его причины — в `DECISIONS.md`, запись от 23.07.2026.

Запрос в поддержку APIShip **не отправлялся**, и отправлять его незачем: партнёрская
программа, autosignup, подключение перевозчиков через `/connections` из интерфейса OCO,
вознаграждение, агентский договор, налоги и НДС — это вопросы Варианта 2 (агентской
схемы), которая отпала вместе с решением. Аккаунт по Варианту 1 заводится как обычным
клиентом. Появится причина спросить — оформим перед запуском.

Осталось **на предзапуск**:

- **ToS APIShip** — гейтит показ продавцу оценочных тарифов по неподключённым
  перевозчикам. Ответ APIShip не нужен: условия опубликованы, их нужно прочитать.
- **152-ФЗ.** После чистки легаси-маршрутов персональные данные в APIShip не уходят
  вообще, поэтому вопрос о поручении на обработку может закрыться сам. Пока чистки нет,
  легаси-создание заказа и синхронизация статусов продолжают отправлять данные получателя
  в APIShip у компаний, подключившихся до 23.07.2026.
