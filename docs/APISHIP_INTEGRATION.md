# APIShip Integration — спецификация (MVP)

Что именно нам нужно от APIShip и как с ним работать. Весь код APIShip живёт ТОЛЬКО в
`packages/integrations/apiship` за единым интерфейсом.

> ⚠️ Важно для Cursor: **точные адреса запросов (endpoints) и форматы полей бери из официальной
> документации** (apiship.ru, docs.apiship.ru), а не из памяти. Ниже — что нам нужно по смыслу,
> а не готовые URL. Если формат поля непонятен — спроси основателя, не выдумывай.

---

## 1. Возможности, которые используем

1. **Расчёт тарифов (calculate)** — на вход: откуда, куда (город/индекс/ПВЗ), вес, габариты,
   объявленная ценность; на выход: список вариантов (служба, услуга, цена, срок).
2. **Список пунктов выдачи (points / ПВЗ)** — для выбора ПВЗ в форме заказа.
3. **Создание заказа (orders)** — на вход: выбранный тариф + данные отправителя и получателя;
   на выход: id заказа в APIShip, трек-номер.
4. **Этикетка/документы (labels)** — получить наклейку для посылки.
5. **Статусы / трекинг (statuses)** — получать обновления статуса по заказу (бесплатно).
6. **Валидация адреса (ApiDQ)** — проверка/нормализация адреса перед заказом (по возможности).
7. **Вызов курьера (courier)** — заявка на забор (по необходимости).

## 2. Единый интерфейс (что должен уметь модуль)

В `packages/integrations/apiship` создать клиент с методами (имена — пример):
- `calculate(input)` → массив вариантов;
- `listPoints(city)` → список ПВЗ;
- `createOrder(input)` → `{ apishipOrderId, trackNumber }`;
- `getLabel(orderId)` → файл/ссылка;
- `getStatus(orderId | trackNumber)` → массив событий;
- `validateAddress(address)` → нормализованный адрес / ошибки;
- `callCourier(input)` → подтверждение.

Остальной код вызывает только эти методы и не знает про устройство APIShip. Это позволит завтра
подменить APIShip на другого агрегатора или прямые договоры, не переписывая продукт.

## 3. Тестовый и боевой контур

- Разработка — **только на тестовом (sandbox) ключе** APIShip.
- Боевой ключ подключаем в самом конце (веха M4), после моего «ок».
- Ключ и базовый адрес API хранятся в `.env` (`APISHIP_API_KEY`, `APISHIP_BASE_URL`), **только
  на сервере**. На фронтенд ключ не попадает никогда — все вызовы APIShip идут через наш бэкенд.

## 4. Что сохраняем из ответов (это актив)

- Из расчёта → **все** варианты в `TariffQuote` (даже невыбранные).
- Из создания заказа → `apishipOrderId`, `trackNumber`, плановые цена и срок в `Shipment`.
- Из статусов → события в `TrackingEvent`, плюс фактические даты и факт возврата/отмены в `Shipment`.

## 5. Устойчивость к сбоям
- Если APIShip недоступен или вернул ошибку — продукт не «падает»: показываем понятное сообщение,
  а создание заказа ставим в очередь на повтор.
- Логируем ошибки **без персональных данных** (без ФИО/телефона/адреса).
- Уважаем лимиты запросов APIShip (не опрашиваем статусы слишком часто — например, раз в N часов).
