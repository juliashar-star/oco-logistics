**Дата:** 2026-07-15 **Что изменилось:** интегрирована выбранная **модель F** (bring-your-own-account, федеративная; APIShip полностью исключён из клиентского пути) из `OCO_altern_2.docx`. План пересобран вокруг двух приоритетов основателя: **(A) сделать максимум ДО переноса на хостинг** и **(B) финализировать максимум ДО передачи юристу** (чтобы юрист получил уже готовые черновики и вопросы, а не переделывал их). **Сведено из:** план v1 (`OCO_edinyy_plan_zadach.md`) · модель F (`OCO_altern_2.docx`) · аудит (severity) · ROADMAP/USER_STORIES/DECISIONS · вопросы юристу. Факты модели F перепроверены (см. §7). **Приоритеты-константы проекта (в порядке важности):** 1) безопасность ИТ и юридическая; 2) удобство и польза клиенту; 3) продуманность, современность, уникальность.

## 0. Как читать

Один **срез = один Cursor-чат**, spec-first, тонкий. ID построены по фазам: `P0-*` (до хостинга и до юриста — делаем сейчас), `P1-*` (нужен хостинг), `P2-*` (нужен юрист/гейт), `P3-*` (позже/не сейчас).

**Статус (§ стратегии):** `[MVP]` · `[позже]` · `[не сейчас]` · `✅ сделано`. **Приоритет:** `P0` блокер запуска · `P1` MVP-критично · `P2` желательно · `P3` позже · `P4` не сейчас.

**Красные линии модели F (соблюдать во всех срезах):**

- OCO — **обработчик** по поручению продавца-оператора; данные **сегментированы по продавцам**, кросс-клиентского пула ПДн нет.
- OCO **вне денег и вне договора перевозки**; не контракт-оф-рекорд. Договоры со службами — у продавца напрямую, на его аккаунтах и ключах.
- **Нейтральность:** никакой платы от перевозчиков за место/видимость/рейтинг. Деньги — только со стороны продавца (подписка + фичи). Допустима лишь разовая плата перевозчика за инженерную работу по его интеграции (за труд, не за позицию).
- Аналитика — только **обезличенная** (Carrier Score без ПДн покупателя); k-анонимность, подавление малых ячеек.
- **Граница APIShip (уточнено):** APIShip/MetaShip исключены из **клиентского пути и исполнения** (клиент их не подключает, доставка идёт через прямые адаптеры на аккаунтах продавца). НО их калькулятор допустимо использовать **как внутренний серверный источник тарифов** для сравнения в слое знания — на мастер-аккаунте OCO, только расчёт (from/to/вес/габариты, **без ПДн покупателя**), невидимо клиенту. Это data-источник, а не труба и не pay-to-rank — нейтральность не нарушается. Проверить ToS APIShip на такое использование.
- Ничего не выдумывать: endpoints и ToS служб — из официальной документации; факты, помеченные «гипотеза», проверять до закладки.

- **Почта — публичный тариф-API** `tariff.pochta.ru`**:** расчёт по общим тарифам, **без договора, без лимита запросов**; `otpravka.pochta.ru` — по договору, со скидками, с лимитами. Годится как публичный источник тарифов для слоя знания.
- **СДЭК:** публичный **веб-калькулятор** без договора есть, но **программный API v2.0 требует логин/пароль интеграции из ЛК (после договора)**. Вывод: тариф СДЭК в рейтинге — только из подключённого аккаунта продавца или из APIShip-бэкфилла (P0-KN7); собственного публичного no-contract API, как у Почты, нет. Не обещать точный API-тариф СДЭК без подключения.
- **152-ФЗ ст.12 (трансграничная передача):** оператор обязан уведомить РКН **до начала** передачи; в «адекватные» страны — можно начинать сразу после подачи уведомления (ждать решения не нужно); для прочих РКН вправе ограничить/запретить в срок рассмотрения. Механика зарубеж-гейта (P0-ZAR1) корректна.
- **Яндекс Доставка — калькулятор-API есть:** `pricing-calculator` проверен живыми вызовами тестового контура 09–10.07.2026. Не требует ни получателя, ни цен товаров — единственный PD-безопасный эндпоинт котировки. Настоящий поток заказа двухфазный: `offers/create` → `offers/confirm` (`request/create` возвращает неподтверждаемые заявки — ловушка). Тестовый контур принимает только московские адреса.

**Гипотезы — проверить ДО закладки (не включать без проверки):**

- **ToS каждой службы и APIShip** на доступ SaaS к API по ключам клиента / использование калькулятора как источника данных (СДЭК, Почта, Яндекс, APIShip).
- Статус **DPD** (собственник/бренд после 2022) — не включать без проверки (урок Boxberry).
- Перечень «адекватных» стран для конкретных направлений (Казахстан/Беларусь/Армения/КНР — в перечне; прочие — уточнить).
- Цена/налоговый режим онбординг-сбора и точка порога **НДС 20 млн ₽** (2026) — сверить с бухгалтером/финмоделью.

## 8. Честные риски модели F (предвижу и предупреждаю)

- **R1 · Регрессия исполнения при уходе от APIShip. → РЕШЕНО.** Ядро M1–M3 работает через APIShip; модель F убирает его из клиентского пути. **Решение основателя (03.07):** пилотируем **слой знания + пост-продажу сразу, БЕЗ исполнения**; прямые адаптеры (Яндекс первым, СДЭК позже) строим по мере готовности/спроса; APIShip не удаляем — держим изолированным dev-fallback (P0-EXE2), сунсет только после боевого СДЭК-адаптера. Тарифы для сравнения (в т.ч. СДЭК, доступный только по договору) до подключения берём из **APIShip/аналог-бэкфилла** на мастер-аккаунте OCO (P0-KN7). Это снимает регрессию: продукт даёт ценность (рейтинг + пост-продажа) ещё до того, как готово собственное исполнение.
- **R2 · Инженерная нагрузка на соло-фаундера — главный риск модели F.** Прямые адаптеры = месяцы суммарно + постоянный «налог обслуживания» на каждую службу + хрупкость (служба может закрыться, как Boxberry). Именно из-за этого APIShip изначально и выбирали. **Митигация:** «знать про всех сразу, возить по спросу» — слой знания стартует с нулём интеграций; адаптеры строить строго по подтверждённому спросу; топ-3–5, не больше; закладывать время на обслуживание.
- **R3 · Не утопить месяцы в адаптеры ДО валидации спроса.** Главный неизвестный проекта — готовность платить, а не архитектура. **Митигация:** сначала дешёвая правда (слой знания + пост-продажа + интервью/пилоты), тяжёлые адаптеры — после первых сигналов спроса.
- **R4 · Поверхность ПДн — экзистенциальна для ИП.** Прямая модель концентрирует ПДн покупателей всех продавцов в одной БД (штрафы 3–15 млн, для ИП — разорение). **Митигация:** обезличенный аналитический слой (разводит моат и штраф), шифрование ПДн at-rest (P0-SEC12, поднят до P1), сегментация по продавцам, минимальный срок хранения, план инцидента 24/72ч.
- **R5 · Трансграничная передача — OCO не «в нуле».** Первичная обязанность уведомления — на продавце-операторе, но OCO как инструмент передачи несёт обязанность по безопасности (ст.19) и договорно-репутационный риск. **Митигация:** зарубеж выключен по умолчанию, гейт (галочка+предупреждение+подтверждение+аудит-след), конструкцию подтвердить у юриста ДО включения.
- **R6 · Нейтральность рушится от денег перевозчиков.** Любая плата за место/рейтинг ломает «независимого подсказчика» (доверие + ст.5 ФЗ «О рекламе»). **Митигация:** деньги только со стороны продавца; онбординг-сбор строго за инженерный труд, отвязан от рейтинга; методика рейтинга опубликована.
- **R7 · Дезинтермедиация вверх по стеку.** APIShip пилит «аналитику тарифов»/«контроль сроков»; Стрекоза/LiveInform — пост-продажу. **Митигация:** моат в связке пост-продажа + обезличенный маховик + вертикаль + system-of-record, не в трубе.
- **R8 · Размер ниши.** DTC-вне-маркетплейсов мал (маркетплейсы 90%+), хотя независимая логистика растёт (+14,5%). **Митигация:** честно валидировать SAM числами; глубина ценности, не широта.

## 9. Что финализируем ДО юриста (сводный чек-лист приоритета B)

Отдать юристу **один раз, готовым пакетом** (срезы P0-DOC1…DOC7), предварительно зафиксировав продуктовые решения P0:

1.  Политика ПДн под модель F (OCO-обработчик, обработчики DaData/Unisender Go, трансграничка, сроки, удаление).
2.  Договор поручения на обработку ПДн (продавец-оператор / OCO-обработчик).
3.  Формулировки согласий в регистрации (разведённые).
4.  Оферта/Условия оплаты (SaaS, онбординг-сбор в УСН, вне денег, не контракт-оф-рекорд, ответственность, возврат).
5.  Cookie-баннер (технические cookie).
6.  Тексты дисклеймеров ЧЗ и зарубеж-предупреждения.
7.  Единый выверенный список вопросов юристу (роль оператор/обработчик, трансграничка, онбординг-сбор vs реклама, k-анонимность, налоги).

Только после готовности пакета — передача юристу (P2-LAW1), затем РКН и публикация.
