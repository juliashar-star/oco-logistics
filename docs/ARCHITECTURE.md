# Architecture — OCO Logistics (MVP)

Как устроен продукт изнутри. Cursor строит структуру по этому файлу. Главный принцип:
**бизнес-логику и интеграции держим отдельно, чтобы менять одно, не ломая другое.**

---

## 1. Общая картина

```
Селлер (браузер)
      │
      ▼
[apps/web] Next.js — экраны + серверные API-роуты
      │
      ├──► [apps/web/lib/shipments] оркестрация заказа: черновик → офферы → отправка
      │            └──► [packages/core/src/carrier-adapter] контракт CarrierAdapter
      │                         └──► yandex/ — пока единственный живой адаптер
      ├──► [packages/core] Decision Engine — правила «Быстро/Дёшево/Оптимально»
      ├──► [packages/integrations/apiship] — слой ЗНАНИЯ (вход Carrier Score и
      │                                      оценочные тарифы), НЕ на пути заказа
      └──► [packages/db] Prisma ──► PostgreSQL (на сервере в РФ)
```

Модель F: продавец везёт по СВОИМ договорам с перевозчиками; OCO — слой оркестрации и
знания, не перевозчик и не агрегатор. Исполнение — ПРЯМЫЕ адаптеры перевозчиков.

- **apps/web** — всё, что видит пользователь, и серверная часть (API-роуты Next.js).
  Здесь же `lib/shipments` — оркестратор заказа: черновик, офферы, отправка, статусы, креды.
- **packages/core** — бизнес-логика выбора доставки и адаптеры перевозчиков. Не зависит от
  Next.js и не знает про экраны, поэтому легко тестируется.
- **packages/core/src/carrier-adapter** — нейтральный контракт `CarrierAdapter` (`types.ts`)
  и реализации, по папке на перевозчика: сегодня `yandex/`, следом CDEK, Почта, 5Post.
  Инвариант: ни один роут и ни один компонент не импортирует `carrier-adapter/<перевозчик>/*`
  напрямую — только через нейтральный контракт. Известный хвост: сейчас инвариант нарушен в
  нескольких роутах, а `types.ts` ещё импортирует типы из `@oco/apiship`; новый код это не
  расширяет.
- **packages/integrations/apiship** — слой ЗНАНИЯ: вход для Carrier Score и оценочный бэкфилл
  тарифов по неподключённым перевозчикам. Никогда не на клиентском пути заказа. Выходим из
  APIShip полностью, когда подключим 10+ перевозчиков напрямую.
- **packages/db** — схема базы (Prisma) и доступ к данным (см. DATABASE.md).
- **packages/shared** — общие типы и утилиты.

**Почему это удобно менять:**
- Поменять вид экрана → только `apps/web`.
- Поменять правило выбора доставки → только `packages/core`
  (`rank-quotes.ts`, `decision-weights.ts`).
- **Добавить перевозчика** → новая папка `packages/core/src/carrier-adapter/<перевозчик>/`,
  реализующая `CarrierAdapter`. Форма, роуты и путь заказа не меняются.
- Поменять или отключить APIShip → только `packages/integrations/apiship`;
  путь заказа не задет.

## 3. Главный поток: «Новый заказ»

1. Селлер заполняет форму посылки в `apps/web` — доставка в ПВЗ или курьером.
2. Создаётся ЧЕРНОВИК: `Shipment` в статусе `DRAFT` (без APIShip и без `TariffQuote`).
3. `POST /api/shipments/:id/offers` → адаптер подключённого перевозчика возвращает офферы.
   Вся выдача сохраняется в `Shipment.quotedOffers`: после подтверждения цена доставки уже
   не восстановима из API перевозчика, а «что нам котировали» — это вход Carrier Score.
4. UI показывает варианты, селлер выбирает. Срок доставки — ОРИЕНТИРОВОЧНЫЙ: перевозчик
   уточняет его позже, а время доставки согласует с получателем сам.
5. `POST /api/shipments/:id/submit` → подтверждение выбранного оффера у перевозчика.
   Сохраняются идентификатор заказа у перевозчика, обещанные цена и срок.
6. Синхронизация статусов читает у перевозчика историю заказа, пишет `tracking_events`,
   фиксирует фактические даты, отмены и возвраты.
7. По фактическим данным пересчитывается Carrier Score.

Старый путь через APIShip (`/api/shipments/calculate` → `TariffQuote` → `/api/shipments/create`)
на живом пути формы не используется. Его код и таблица `TariffQuote` остаются до отдельной
чистки легаси — не строй на них новое.

## 4. Технологии

Frontend/Backend: Next.js (App Router) + TypeScript + Tailwind + shadcn/ui.
База: PostgreSQL 16, ORM Prisma. Запуск: Docker Compose. Хостинг: российский сервер.
Код и имена — на английском; комментарии и общение с основателем — на русском.

## 5. Безопасность и закон (кратко; детали — в агенте «Безопасность»)
- Персональные данные получателей — только в нашей базе на сервере в РФ, в зашифрованном виде.
- ПДн не пишем в логи, URL, ошибки.
- Токены и логины перевозчиков лежат не в `.env`, а пошифрованно в базе, отдельной записью
  на компанию (`CarrierCredential`). Ключ APIShip — один мастер-аккаунт OCO, в `.env` на сервере.
- Секреты не попадают в git. HTTPS обязателен в бою.
