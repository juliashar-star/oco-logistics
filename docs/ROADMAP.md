# Roadmap — OCO Logistics (MVP)

План по вехам. Каждая веха — самостоятельный кусок, который можно показать и проверить.
Сроки указаны для работы в одиночку с Cursor (ориентир — 6–8 недель). Cursor использует
этот файл, чтобы понимать порядок работ и не забегать вперёд.

Правило: **не начинаем следующую веху, пока предыдущая не проверена руками.**

---

## M0 · Каркас (≈ неделя 1)
Цель: пустой, но рабочий скелет проекта.
- Создать структуру папок (см. ARCHITECTURE.md).
- Поднять Next.js + Prisma + PostgreSQL в Docker Compose локально.
- Открывается пустая главная страница, приложение подключается к базе.
- Готов `.env.example`, базовая авторизация (заглушка входа).
**Готово, когда:** локально открывается кабинет, база подключена, `docker compose up` работает.
Истории: US-1.1, US-1.2 (база).

## M1 · Ядро «Новый заказ» (≈ недели 2–3)
Цель: главная ценность продукта работает на тестовом APIShip.
- Слой интеграций APIShip (sandbox): расчёт тарифов, список ПВЗ.
- Форма посылки + таблица вариантов.
- Decision Engine: «Быстро/Дёшево/Оптимально».
- Создание отправления → трек-номер.
- Сохранение всех вариантов и отправления в базу (актив с первого дня).
**Готово, когда:** ввёл посылку → увидел варианты → создал заказ → получил трек, всё за < 3 минут.
Истории: US-2.1, US-2.2, US-3.1–US-3.4.

## M2 · Отправления и трекинг (≈ недели 4–5)
Цель: контроль над уже созданными отправлениями.
- Список отправлений с фильтрами и поиском.
- Периодический опрос статусов APIShip, сохранение событий трекинга.
- Фиксация фактических дат, возвратов, отмен.
- Экспорт в CSV.
**Готово, когда:** все заказы видны со статусами, статусы обновляются, история событий сохраняется.
Истории: US-4.1–US-4.4.

## M3 · Данные и дашборд (≈ недели 6–7)
Цель: продукт начинает «умнеть» и показывать пользу.
- Carrier Score по накопленным данным.
- «Оптимально» учитывает Carrier Score.
- Дашборд: заказы, расходы, экономия, график.
- Онбординг на пустых экранах.
**Готово, когда:** дашборд показывает реальные цифры, рейтинг влияет на рекомендации.
Истории: US-5.1–US-5.3.

## M4 · Боевой запуск (≈ неделя 8)
Цель: можно пускать реальных клиентов — по закону и безопасно.
- Развернуть на российском сервере (см. инструкцию по подключению сервера).
- Домен + HTTPS, ежедневные бэкапы базы.
- Переключить APIShip на боевой ключ.
- Разместить Политику обработки ПДн и согласие; зарегистрироваться оператором ПДн в РКН;
  добавить IT-ОКВЭД в ИП.
- Подключить первые 1–2 пилотных бренда (часть заказов сначала можно вести вручную — concierge).
**Готово, когда:** пройден чек-лист «можно пускать первого клиента».

## Что параллельно (не код, но критично)
С первого дня — интервью с брендами (30–50) и подбор 5 пилотных. По документу это важнее кода:
код сегодня дешёвый, а первые бренды и их данные — нет.

## После MVP — направление развития (Phase 2)
НЕ строим сейчас; это записано, чтобы строить MVP с прицелом на будущее (см. docs/DECISION_ENGINE.md).
Браться за это — только после того, как MVP подтверждён на пилотных брендах.
- Правила автовыбора перевозчика по условиям (вес, габариты, тип доставки, регион, категория) —
  с интерфейсом настройки для продавца.
- Автоматический режим без ручного выбора для крупных продавцов (сотни заказов в день):
  пакетная обработка, фоновая очередь, идемпотентность, учёт лимитов APIShip.
- API для продавцов и массовый импорт заказов — два разных канала: (а) собственный сайт/магазин
  продавца (Tilda, InSales, свой движок и т.п.), который передаёт заказ в OCO через API/вебхук без
  ручного ввода; (б) интеграции с маркетплейсами и массовый CSV-импорт. Для MVP продавец всегда
  вводит адрес сам, вручную, в кабинете — на основе того, что покупатель указал в его канале продаж.
- Публичная страница/ссылка для покупателя (без входа в систему), где он сам вводит свой адрес
  и желаемое время доставки, минуя продавца. Снижает ручной труд при больших объёмах, но требует
  отдельной защищённой публичной формы, защиты от спама и канала доставки ссылки (SMS/мессенджер) —
  это отдельный продуктовый кусок, не часть логистического MVP.
- Расширенное использование полей APIShip (страховка, объявленная ценность, наличный расчёт и др.).
- Коэффициенты второго уровня: качество по региону и категории, стабильность срока,
  доля повреждений/потерь, штраф за просрочку.
