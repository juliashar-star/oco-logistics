# Roadmap — OCO Logistics (MVP)

План по вехам. Каждая веха — самостоятельный кусок, который можно показать и проверить.
Сроки указаны для работы в одиночку с Cursor (ориентир — 6–8 недель). Cursor использует
этот файл, чтобы понимать порядок работ и не забегать вперёд.

Правило: **не начинаем следующую веху, пока предыдущая не проверена руками.**

---

## M0 · Каркас (≈ неделя 1)
Цель: пустой, но рабочий скелет проекта.
- Создать структуру папок (см. ARCHITECTURE.md).
- Поднять Next.js + Prisma + PostgreSQL в Docker Compose локально.
- Открывается пустая главная страница, приложение подключается к базе.
- Готов `.env.example`, базовая авторизация (заглушка входа).
**Готово, когда:** локально открывается кабинет, база подключена, `docker compose up` работает.
Истории: US-1.1, US-1.2 (база).

## M1 · Ядро «Новый заказ» (≈ недели 2–3)
Цель: главная ценность продукта работает на тестовом APIShip.
- Слой интеграций APIShip (sandbox): расчёт тарифов, список ПВЗ.
- Форма посылки + таблица вариантов.
- Decision Engine: «Быстро/Дёшево/Оптимально».
- Создание отправления → трек-номер.
- Сохранение всех вариантов и отправления в базу (актив с первого дня).
**Готово, когда:** ввёл посылку → увидел варианты → создал заказ → получил трек, всё за < 3 минут.
Истории: US-2.1, US-2.2, US-3.1–US-3.4.

23. **Reaper для зависших заказов**: submitOrder помечает «неизвестно»
    как PROBLEM (сеть легла на confirm, либо запись упала после успешного

24. **Валидация offer.deliveryIntervalFrom до confirm**: кривая ISO-строка

    заказа у перевозчика.
25. **Вызывающая сторона submitOrder**: он может И вернуть {ok:false,…},

    Будущий submit-роут обязан оборачивать вызов в try/catch, а не только
    читать результат.
26. **Актуализация документов (doc-drift)**: мастер-план (docx) говорит

## M5 · После запуска (backlog)
Цель: доработки и перевозчик-специфичные возможности, отложенные до стабильного M4.
Не начинаем, пока M4 не проверен руками и пилоты не работают.

- Автоматический расчёт объёма отгрузки
  - Объём вычисляется автоматически из габаритов мест (длина × ширина × высота), не вводится вручную.
  - Для MVP — сумма объёмов мест (коробок), а не товаров. Учёт упаковки и пустот — отдельно, позже.
  - Контекст: некоторые перевозчики (например, Яндекс.Доставка) требуют объём в м³ при создании отгрузки.

- Поле фото товара (photoUrl) и проброс фото перевозчику
  - Добавить опциональное поле photoUrl у товара (публичная ссылка на изображение, валидный URL до 2048 символов).
  - Проброс фото — перевозчик-специфичная опция. На текущий момент в документации APIShip передача фото описана только для Яндекс.Доставки (поле item.photoLinks → photo_links, только тарифы «Доставка в другой день», только при создании заявки, не при редактировании, не для Cargo/Express).
  - Для других перевозчиков поддержку проверять индивидуально при подключении.

- Слой перевозчик-специфичных полей при создании отправления
  - Универсальная форма создания отправления не покрывает все случаи: у каждого перевозчика свои дополнительные поля и опции (у Яндекса — планирование отгрузки, объём/вес отгрузки, опция «нужна помощь с погрузкой», повтор по дням недели; передача фото; и т.п.).
  - Нужен слой, который добавляет перевозчик-специфичные поля поверх базовой формы.
  - Это архитектурное решение, проектировать отдельно после MVP.

## После MVP — направление развития (Phase 2)
